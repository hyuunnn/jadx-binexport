# CLAUDE.md — jadx-binexport / jadx → BinExport plugin

Working notes for continuing development. Reads best top-to-bottom once, then as a reference.

## What this is

A **jadx plugin** that exports jadx's analysis (classes, methods, CFG, call graph)
into the **BinExport2 protobuf** consumed by **BinDiff**, so two APKs' Dalvik
bytecode can be diffed/visualized in BinDiff. The plugin does *pure mapping*: jadx
already computes everything (methods, basic blocks, dominators, call refs); we
serialize it. The proto was designed with DEX in mind (`Architecture::kDex`,
`Module` = Java class, `Vertex.module_index`), so this is an intended use, not a hack.

src/main/java/dev/jadxbinexport/

Plugin id: `jadx-binexport`. Output jar: `build/libs/jadx-binexport-<ver>.jar`.

## Install / run

```bash
jadx plugins --install-jar build/libs/jadx-binexport-0.1.0.jar
jadx -d out_v1 app-v1.apk        # -> out_v1/app-v1.BinExport
jadx -d out_v2 app-v2.apk        # -> out_v2/app-v2.BinExport
```
Output path resolution (first wins): plugin option `jadx-binexport.output` (legacy `-Dbinexport.output`) → `jadx-binexport.outdir` (legacy `-Dbinexport.outdir`) → jadx out dir, always resolved to an **absolute** file (jadx-gui leaves the out dir cwd-relative, which for a Finder-launched app is `/`). Existing files are overwritten with a log warning. GUI: `Plugins → Export to BinExport`, which then shows a dialog with the resolved output path (or the failure).

## GOTCHAS (hard-won — read before touching Exporter)

1. **`cls.decompile()` DESTROYS the CFG.** `ProcessClass.process(cls, codegen=true)` calls `cls.unload()` right after code generation, nulling `MethodNode.blocks`. Symptom: export runs but `instructions=0`, empty flow graphs. Fix in use: `cls.root().getProcessClasses().forceProcess(cls)` runs the decompile passes (block split, SSA, dominators, region maker) **without** codegen/unload, leaving blocks intact and state `PROCESS_COMPLETE`. Normal `jadx.save()` afterwards still works (codegen from the already-processed state).

2. **jadx has no "after decompile" hook.** Pass types are only `JadxPreparePass`, `JadxDecompilePass`, `JadxAfterLoadPass`. `JadxAfterLoadPass.init(decompiler)` fires at the *end of `load()`*, **before** any decompilation. So the pass must drive decompilation itself (that's what `forceProcess` above is for) and then write the file. We use jadx's `SimpleAfterLoadPass` (no custom pass class needed). There is no streaming "last method" signal — the after-load-drives-everything design gives a clean single-threaded pass with a definite end.

3. **GUI vs CLI branching.** `context.getGuiContext() != null` ⇒ GUI: register on-demand menu actions (don't auto-export the whole app on every project open). Else CLI/library: register the AfterLoadPass calling `Exporter.runLogged(decompiler, options)` (single error contract, slf4j — `System.out` never reaches jadx-gui's Log Viewer). The GUI actions fetch `context.getDecompiler()` at **click time** (the init-time instance can go stale on project reload). Concurrency is guarded at **two levels**: the Export action's `AtomicBoolean` gives fast "already running" feedback and protects its single output file; **`Exporter.EXPORT_LOCK`** (a static `ReentrantLock` that both `run` and `runToFile` acquire via `exportLocked`) serializes the actual model mutation so the Export action, the Open-BinExport diff action, and any two diffs can never drive `forceProcess` over the same jadx model concurrently (jadx's decompile passes are not thread-safe). The GUI Export action calls `Exporter.run` directly (not `runLogged`) so it can show a success dialog with the absolute output path / an error dialog — a log-only failure is invisible without the Log Viewer, and the GUI has no out dir configured so the resolved path is not obvious.

3c. **Progress + cancel (both GUI actions).** A big app takes minutes, so the GUI shows an `ExportProgressDialog` (Swing bar + Cancel), like the IDA BinExport wait box / Ghidra `TaskMonitor`. The plumbing is a headless `ExportProgress` interface (`stage`/`update`/`cancelled`, all no-op defaults so callers/test-doubles override only what they use; `ExportProgress.NONE` = `new ExportProgress(){}`, `ExportProgress.orNone(p)` coalesces null once at each entry point), threaded through `Exporter.run`/`runToFile`, `BinDiffRunner.diff`, and `BinDiffResults.methodIndex`; the CLI/library path passes `NONE`, keeping `Exporter` free of any Swing dependency and headless-testable.

  **Cancel must leave nothing behind, so it is polled at EVERY phase, not just the two big loops.** The export runs `Decompiling classes` → `Building flow graphs` → `Building call graph` → `Writing file`; a private `tick(done,total,mask)` polls `cancelled()` and posts a **throttled** `update` (every 64 classes / 256 methods, advancing to `done+1` and flushing the final item so the bar reaches the total) inside the big loops, and `checkCancelled()` gates every phase boundary INCLUDING the one right before `writeTo` — so a cancel at any point up to the write throws `CancelledException` and no file is written (the earlier "only the two loops were checked, so a late cancel still wrote the file" bug). A queued export/diff waiting on `EXPORT_LOCK` polls cancel via `tryLock(200ms)` instead of blocking uninterruptibly. In the diff, `BinDiffRunner.runProcess` polls `cancelled()` in 200ms slices while bindiff runs and `destroyForcibly()`-kills it on cancel — but it checks the process **exit before** the cancel/deadline each slice, so a bindiff that already finished is never discarded or misreported as a timeout (overflow-safe `deadline - nanoTime() <= 0`); on kill the reader thread's IOException is swallowed (debug log), not dumped to stderr. `promptAndDiff` also polls cancel around the result load and passes the sink into `methodIndex`'s class walk. `runLogged` and both GUI actions catch `CancelledException` distinctly from a real failure (info log, no error dialog).

  **Dialog** is thread-safe by construction: all Swing access is marshalled to the EDT via `invokeLater` (FIFO-ordered after the constructor's build post); `cancelled()` reads a `volatile` the Cancel button sets on the EDT; the title-bar **X is wired to request cancel** (`DO_NOTHING_ON_CLOSE` + `windowClosing`), never HIDE_ON_CLOSE (which would hide the dialog while work ran on invisibly). The GUI export builds the dialog **inside** the try that resets the `running` AtomicBoolean, and `promptAndDiff` closes the dialog in a `finally` and guards `thread.start()`, so neither a dialog-ctor throw nor a thread-start failure can wedge the menu or leak the dialog.

3a. **Use `getRoot().getClasses()`, NOT `decompiler.getClasses()`.** The latter filters out classes flagged `AFlag.DONT_GENERATE` (deduplicated multidex entries, synthetic holders) — their methods would silently vanish from the call graph, producing phantom diffs. `getRoot().getClasses()` enumerates the full model (including inner classes; the visited-set handles the overlap with the inner/inlined recursion).

3b. **IF edge labels come from `IfNode.getThenBlock()/getElseBlock()`, NEVER successor order.** jadx connects the fall-through (else) successor first, and `invertCondition()` swaps then/else *without* reordering successors — "first successor = TRUE" is inverted in the common case. Verified against jadx 1.5.6 sources.

4. **jadx IR is normalized, NOT raw smali.** Instructions are jadx's post-transform IR (`InsnType`), and many `invoke`s get **folded into operand trees** as `InsnWrapArg`. Consequences:
   - Call-graph completeness requires recursing wrapped args (`arg.isInsnWrap()` → `arg.unwrap()`), else you miss calls. `Exporter.collectCallees` does this, and `emitInstruction` runs it per instruction so folded invokes also get `call_target` — callees are accumulated per method there, so the call graph and per-instruction data can never disagree (single IR sweep, no separate call-graph pass). Self-recursion edges are **kept** (BinDiff's degree-based matching expects them).
   - Two files are only comparable in BinDiff if produced by the **same jadx version** (mnemonic set + folding differ across versions). This is a hard rule, surface it to users.

5. **Empty/synthetic blocks.** jadx enter/exit blocks often have 0 instructions. A `BasicBlock` needs ≥1 instruction for its index range, so we emit a synthetic `nop` for empty blocks (kept in the mnemonic histogram too). Don't "optimize" these away or CFG edges break.

6. **Degraded-body accounting.** `resolveBlocks` decides ONCE which methods export with a body; all three passes share `blocksByMethod`. A method exports as a bodyless call-graph vertex when: it has no code; its top-level class failed `forceProcess` (partially-transformed IR would export misleading bodies — failures are counted and summarized at the end); or its emitted instruction count would spill past the 2^20 per-method address stride (addresses would stop being unique — logged per method). Bodyless methods **still contribute call-graph edges** — `buildCallGraph` walks their `rawBlocks` for callees; only the flow graph/instructions are withheld. Duplicate `getRawFullId()`s (same class in two inputs / deduplicated multidex entries) are collapsed to the **first** copy in `export()` step 2 *before* addresses are assigned: the second copy is dropped from the `methods` list entirely (not just from call-target resolution), so we never emit two call-graph vertices with an identical `mangled_name` at different addresses — BinDiff's name matching (its default step #1) cannot tell such copies apart and would pair them arbitrarily. Calls to that id resolve to the surviving first copy; the drop is logged per id.

7. **Enter block must be emitted FIRST.** BinDiff resolves a flow graph to its function via the entry block's first-instruction address == vertex address. jadx normally keeps the enter block at `getBasicBlocks().get(0)`, but `BlockProcessor.insertPreHeader` (method body starting with a loop) creates a fresh enter block via `startNewBlock`, which APPENDS to the list. `resolveBlocks` reorders the enter block to the front before emission-order address assignment; without it BinDiff throws `std::runtime_error("couldn't find call graph node for flow graph")` and aborts the entire diff.

8. **Synthetic `raw_bytes` are load-bearing for obfuscated diffs.** We emit a canonical text rendering of each insn tree as `raw_bytes`. BinDiff SDBM-hashes these into its function/basic-block "hash matching" steps — its two highest-confidence matchers. With empty bytes every hash is 0 and those steps can never match. On same-name inputs the effect is small (name matching runs first), but when ProGuard/R8 renames symbols between versions — the primary use case — content hashes become the top signal. The rendering (`renderInsn` + `appendPayload`) covers mnemonic, callee id, result/arg regs, literals, wrapped insns **and the payloads that live in InsnNode *subclass* fields, not in the argument list**: `ConstStringNode.getString()`, `ConstClassNode.getClsType()`, `IndexInsnNode.getIndex()` (IGET/IPUT/SGET/SPUT/CHECK_CAST/NEW_INSTANCE field or type ref), `IfNode.getOp()` and `ArithNode.getOp()` (the mnemonic alone renders every `if`/`arith` identically), `SwitchInsn.getKeys()`. These are exactly the per-method constants that break MD-index ties — omitting them collapsed two methods differing only in a constant or operator to identical bytes. Rendering must stay deterministic and file-independent (payload *content* only — never a file-local operand/expression INDEX). NB: enriching the rendering changes every SDBM hash, so a `.BinExport` is only comparable against one produced by the **same plugin version** (already true across jadx versions).

8a. **In-GUI diffing (IDA-like), the second GUI menu action.** A diff is always made from two `.BinExport`s; jadx already holds one (the open app), so the user only opens the OTHER side's export. "Open BinExport (.BinExport)…" is the one-step flow: `BinDiffRunner.diff` exports the current app into a temp work dir (`Exporter.runToFile`), shells out to the `bindiff` executable, copies the produced `.BinDiff` out to a standalone `deleteOnExit` temp file, and (in a `finally`) **deletes the whole work dir** — it holds a multi-MB `current.BinExport`, so leaving it leaks unbounded temp disk across a session. The `.BinDiff` feeds straight into the results table. The matching engine is bindiff itself — we never reimplement it. Deliberately there is **no** "open a raw `.BinExport` viewer" (it carries no source) and **no** "open a pre-made `.BinDiff`" action — the diff is regenerated from the two exports on demand, which subsumes it. `promptAndDiff` runs the whole flow — export → bindiff → read `.BinDiff` → `show` the table — on **one** worker thread driving a single `ExportProgressDialog` (stages: "Running BinDiff…" indeterminate, then "Loading results…"); the diff/export failures and a user cancel are handled there. (The old separate `loadAndShow` second thread was folded in — one thread, one error site.) A `DIFF_RUNNING` `AtomicBoolean` rejects a second concurrent diff (mirrors the Export action's guard). The returned `.BinDiff` is **fully read during the load** (`loadMatches`/`loadHeader`/`methodIndex`) and `show` only needs its *name*, so the worker's `finally` **deletes it eagerly** (and clears the guard) rather than relying on the `deleteOnExit` backstop — otherwise every diff would leak a ~20MB temp file for the whole session. `loadMatches` and `methodIndex` take the progress sink and poll `cancelled()` mid-read/mid-walk so "Loading results…" is cancellable, not just discardable.

  **Subprocess handling** (`BinDiffRunner.runProcess`, shared by the diff and the discovery probe): output is drained on a **separate thread** (a full pipe buffer must not block the child), decoded as **UTF-8** (never the platform default), and bounded by a **timeout** — `waitFor(timeout)` then `destroyForcibly()` — so a wedged bindiff can't hang the GUI thread forever (30 min for a diff, 20 s per `--help` probe). A non-zero exit **with** a `.BinDiff` present is logged as a warning (partial/truncated DB), not silently treated as success.

  **bindiff discovery** (`resolveBindiff` → `findBindiff`/`probe`): an explicitly configured `jadx-binexport.bindiff` is **authoritative** — if it doesn't run, fail with a message naming the option, never silently fall back to a different bindiff on PATH. Only when it's unset do we auto-detect via PATH then common install paths, which **include `/opt/homebrew/bin/bindiff`** (Apple-Silicon Homebrew — a Finder-launched jadx-gui inherits launchd's minimal PATH excluding it). bindiff is still detected by `--help` OUTPUT containing "bindiff" since it exits non-zero (Google-flags CLI).

9. **Results table + navigation.** `promptAndDiff` reads the `.BinDiff` and shows a navigable table; double-click => `JadxGuiContext.open(methodNode)` jumps to the method. Key points: (a) `.BinDiff` is a **SQLite** file — its `function` table has `name1/name2/similarity/confidence`, and `name1/name2` are exactly our `mangled_name` (`MethodInfo.getRawFullId()`), so navigation is **by name, not by the synthetic address**. (b) `BinDiffResults.methodIndex` MUST enumerate methods the SAME way `Exporter` does (`getRoot().getClasses()` + inner/inlined, includes `<init>`) — using the API-level `JavaClass.getMethods()` drops constructors and misses matches. Enforced in code: both sides recurse via the shared `Exporter.walkClassTree` (identity visited-set + inner + inlined), so the enumeration cannot drift. (c) `MethodNode implements ICodeNode extends ICodeNodeRef`, so it is passed straight to `open()`. (d) SQLite via `org.xerial:sqlite-jdbc`, **bundled but NOT relocated** (jadx has no SQLite so no clash; relocating `org.sqlite` breaks its native-lib resource lookup). Use `SQLiteDataSource` (not `DriverManager`) to dodge the child-classloader SPI problem in the plugin classloader. Testable core (`BinDiffResults`) is separate from the Swing shell (`BinDiffResultsPanel`) because Swing can't instantiate headless. (e) `SimilarityRenderer` tints the similarity cell red→green over a fixed near-white background, so it **pins `setForeground(Color.BLACK)`** — a dark theme's default table foreground is near-white and would render white-on-white otherwise.

10. **Sysprop passing on the jadx CLI.** jadx has NO `-J` JVM-arg passthrough (unknown options become input files via JCommander's `acceptUnknownOptions`). Legacy sysprops must go through `JADX_OPTS`/`JAVA_OPTS` env vars; the plugin options (`-Pjadx-binexport.output=...`) are the primary interface. jadx calls `setOptions` during `registerOptions()` in every mode (CLI/GUI/library), and `BasePluginOptionsBuilder` invokes setters with `defaultValue` for absent keys.

## BinExport2 format rules (from the canonical reference)

Canonical reference is Google's Ghidra exporter `java/src/main/java/com/google/security/binexport/BinExport2Builder.java` in `google/binexport`. Mirror it, not guesses.

- **FlowGraph indices are GLOBAL.** `entry_basic_block_index`, `basic_block_index`, and edge `source/target_basic_block_index` all index into the top-level `BinExport2.basic_block` array — NOT into the flow graph's local list.
- **Mnemonic index 0 = most frequent** (build a histogram, sort desc by count then name). Omit `mnemonic_index` when it's 0.
- **BinDiff consumer contract** (verified against google/bindiff source, `flow_graph.cc`/`call_graph.cc`/`match/*`): hard rules are (1) call-graph vertices sorted by address, (2) each flow graph's blocks sorted by first-instruction address, (3) every BasicBlock has ≥1 instruction range (`CHECK`), (4) valid edge enum, (5) **flow-graph entry address == a call-graph vertex address** or `AttachFlowGraph` throws and aborts the diff (why we force the enter block to the front, see gotcha 8). Matching consumes ONLY: mnemonic strings (prime products + LCS), `raw_bytes` (SDBM hashes → the two top-confidence "hash matching" steps), `call_target`, mangled/demangled names (name hash matching is default step #1), and graph topology. **Ignored by matching**: operands/expressions, string_table/`string_reference` (BinDiff hardcodes `string_hash_=0`, TODO in their code), CONDITION_TRUE/FALSE (display only), `is_back_edge` (never read; BinDiff recomputes loops itself), meta_information (display only). BinDiff silently discards function bodies ≥10k instructions / ≥5k blocks / ≥5k edges (we export them anyway and log a count).
- **Vertices MUST be sorted by ascending `address`** (BinDiff requirement). We add them in method-index order and give method `i` address `(i+1) << 20`, so ordering is automatic; `vertexIndex == methodIndex` for in-app methods. With the opt-in `imports` option, IMPORTED stubs are appended at vertex indices `methods.size()+j` with address `((methods.size()+j)+1)<<20` — strictly above every in-app address, so the list stays sorted (see backlog 1).
- `Instruction.address` is optional (fill only on discontinuity); we fill it on **every** instruction (simpler, valid).
- `Expression` default type is `IMMEDIATE_INT` → leave `type` unset for immediates (matches "omit defaults" intent).
- De-dup `Expression`/`Operand` keyed by the primitive that fully determines them (literal long / reg num / symbol string — every operand here is a single expression), so the hit path allocates no throwaway protos; same result as BinExport's proto-keyed scheme. Instructions/blocks are NOT deduped — we emit them method-by-method so each block's instructions are a contiguous `[begin,end)` range. Instructions/blocks/flow-graphs are added as **built messages** (`addInstruction(msg)`), not via `addXxxBuilder()` — repeated-field builder mode would hold millions of mutable wrappers in memory.

## Address synthesis

Dalvik has no linear address space. Scheme: sort methods by `MethodInfo.getRawFullId()` (cached per method — jadx rebuilds the string on every call), method `i` → base `methodAddress(i) = (i+1) << 20`; instruction address = `base + sequenceIndex`. `1<<20` (~1M) headroom per method; methods that would exceed it are exported bodyless with a warning (see gotcha 6). `methodAddress()` is the **single** helper linking vertex addresses, instruction bases and `call_target` — never inline the formula. Addresses only need to be **unique + stable within one file** (BinDiff matches structurally, not by absolute address); stable ordering keeps the same method at the same address across rebuilds of the same input.

## Key jadx API cheat-sheet

- `JadxDecompiler.getRoot().getClasses()` → full `List<ClassNode>` incl. inner + `DONT_GENERATE` (what we use). `decompiler.getClasses()` → filtered `List<JavaClass>` (drops `DONT_GENERATE`; don't use for export).
- `ClassNode`: `root()`, `getMethods()`, `getInnerClasses()`, `getInlinedClasses()`, `getClassInfo().getFullName()`.
- `MethodNode`: `getMethodInfo()`, `isNoCode()`, `getBasicBlocks()` (null/empty until processed), `getEnterBlock()`, `getParentClass()`.
- `MethodInfo`: `getRawFullId()` (stable, `$`-separated inner), `getFullId()` (readable), `getShortId()`.
- `BlockNode`: `getInstructions()`, `getSuccessors()`, `getPos()`, `getDoms()` (BitSet) — back edge ⇔ target dominates source: `from.getDoms().get(to.getPos())`.
- `InsnNode`: `getType()` (`InsnType`), `getResult()`, `getArguments()`, `getOffset()`. Payload-bearing subclasses (fields NOT in `getArguments()`, rendered into `raw_bytes` by `appendPayload`): `ConstStringNode.getString()`, `ConstClassNode.getClsType()`, `IndexInsnNode.getIndex()`, `IfNode.getOp()`, `ArithNode.getOp()`, `SwitchInsn.getKeys()`.
- Calls: `insn instanceof BaseInvokeNode` → `getCallMth()` (`MethodInfo`). Args: `InsnArg.isLiteral()/isRegister()/isInsnWrap()`, `LiteralArg.getLiteral()`, `RegisterArg.getRegNum()`.

## Test approach

`ExporterIntegrationTest` compiles a small class with `ToolProvider` javac, loads it through jadx via **`jadx-java-input`** (compiled `.class` path, not DEX), lets the AfterLoadPass export, then `BinExport2.parseFrom` and asserts invariants (sorted vertices, entry∈blocks, global indices in range, valid instruction ranges, ≥1 call edge, a back edge from the loop, CONDITION_TRUE **and** CONDITION_FALSE edges present, a self edge for the recursive `fact()`, ≥1 instruction with `call_target`). Last run: `vertices=5 edges=3 instructions=37 basicBlocks=32 flowGraphs=5 mnemonics=6 backEdge=true`. A second method in the same class drives the progress path headlessly: a recording `ExportProgress` must see the "Decompiling classes" and "Building flow graphs" stages with in-range counts, and a sink whose `cancelled()` returns true must make the export throw `Exporter.CancelledException` and write **no** file.

Opt-in tests (skip cleanly when their tool/display is absent):
- `RealApkSmokeTest` — real DEX path + determinism. Needs a real APK: `./gradlew test -Pbinexport.smoke.apk=/path/app.apk`. That flag also bumps the test worker to `maxHeapSize='4g'` — a big app builds one large in-memory `BinExport2` (hundreds of thousands of builder messages before serialization) and OOMs the default ~512MB worker heap; jadx's own CLI/GUI run with a large heap, so this just matches the shipping runtime.
- `BinDiffNavigationTest` / `BinDiffRunnerTest` — need the `bindiff` CLI (found via PATH / `-Dbinexport.bindiff` / common paths; detected by `--help` OUTPUT since it exits non-zero). They run the real export→bindiff→read-results→resolve-to-MethodNode loop.
- `BinDiffClickTest` — needs a non-headless display (`-Djava.awt.headless=false`); builds the real results table and dispatches a genuine double-click MouseEvent, asserting navigation targets the right `MethodNode`. Swing can't run headless, hence the split between the Swing shell (`BinDiffResultsPanel`) and its testable core (`BinDiffResults`).

  - **raw_bytes ON: 50/52 correct (96.2%)** — matches came from flow/call-graph MD-index (31) + `function: hash matching` (15, the step that consumes raw_bytes); name matching only 1. The 2 "wrong" are a genuinely-new v2 method paired to a structurally similar one (no correct answer exists).
  - **raw_bytes OFF (A/B control): 12/52 correct (23.1%)** — MD-index steps collapse (the sample's methods share a near-identical loop/branch skeleton, so topology MD indices collide and can't disambiguate) and BinDiff falls back to `function: address sequence` (39), which is meaningless across two independent obfuscations.
  - Conclusion: on obfuscated, structurally-repetitive code, `raw_bytes` (which encodes the per-method constants/operands that break MD-index ties) is the difference between **96% and 23%** correct matching. Not a marginal nice-to-have.
